"Update the spec first."

I must have said that to Claude about fifteen times during a single session. And every time I forgot to say it, the agent forgot too. We'd fix a bug, ship a feature, refactor a composable - and the specification would quietly drift out of sync with reality.

That's when I realized: **the hardest part of building with AI agents isn't the code. It's the discipline.**

***

## The Setup

I'm building [SpeQA](https://barsia.github.io/speqa) - an IntelliJ IDEA plugin for manual QA testers. Test cases stored as Markdown files with YAML frontmatter, edited through a custom split editor with Compose/Jewel UI. The kind of project where the spec matters because the file format, the parser, the serializer, the editor UI, and the validation layer all need to agree on what a test case looks like.

I started the way most people start with AI coding agents: "Let's brainstorm this feature." Claude spun up a brainstorming skill, asked me questions, proposed approaches, and we landed on a design. Clean process. Good output.

Then came the first real lesson.

## The Rule That Changes Everything

Before writing a single line of code, I created an `AGENTS.md` file with three non-negotiable rules:

```markdown
## Required workflow

1. Start by editing the PRD/specification.
2. The PRD/specification must always describe the current product state.
3. After updating the PRD/spec, change the code so the implementation matches it.
```

This isn't just documentation hygiene. It's an **architectural decision about how the agent works**.

Without these rules, here's what happens: you ask the agent to fix a bug. It fixes the bug. The spec still describes the old behavior. Next time the agent reads the spec, it thinks the old behavior is correct. Now you have a spec that actively misleads the agent that reads it.

With these rules, the spec becomes the single source of truth - not the code, not the git history, not the agent's memory. The spec.

## A Change Cycle in Practice

Here's what a proper change cycle should look like — and the moment from our session that taught us why:

**User:** "Make Priority and Status optional fields."

**What the agent should do:**

1. Update the spec: change "required" to "optional", add field presence rules
2. Update the data model: `Priority` → `Priority?`
3. Update the parser: check `"priority" in meta` before parsing
4. Update the serializer: skip null fields
5. Update the UI: render empty state for null fields
6. Update tests

**What the agent actually did the first time:**

1. Update the data model
2. Update the parser
3. Update the serializer
4. Update the UI
5. Update tests
6. *...forget the spec entirely*

I caught it. "Did you update the spec?" And from that point on, I saved a memory: *Always update spec before code changes. This is a hard rule.*

The agent learned. Sort of. It still needed reminding, but less often. And the `AGENTS.md` file was there as a permanent instruction - not just for this session, but for every future session.

## The AGENTS.md as Living Architecture

`AGENTS.md` isn't a README. It's the agent's operating manual. Ours grew organically during the session:

```markdown
## Engineering rules

- Fix root causes, not symptoms.
- Do not break IntelliJ platform expectations.
- Keep plugin.xml, file templates, actions, icons consistent.
- Do not add non-English text in code. Only in localization files.
- Never hardcode user-visible strings in Kotlin code.
  All UI text must be in SpeqaBundle.properties.
```

That last rule was added mid-session when I noticed hardcoded strings everywhere. One conversation, one `AGENTS.md` update, and every future agent invocation knows the rule.

This is the real power: `AGENTS.md` **is institutional knowledge that survives context windows.**

## The PRD as a Living Document

If `AGENTS.md` captures *how* we work, the PRD captures *what* we're building. Both need to be living documents.

My spec started at ~100 lines. By the end of the session, it was ~500 lines. Not because we added fluff - because we kept refining decisions.

Here's how the test case file format evolved:

**Version 1** (brainstorm output):

```markdown
### Step 1: Enter username
**Action:** Type "test@example.com"
**Expected:** Email field accepts input
```

**Version 2** (after feedback on Version 1 - "too verbose"):

```markdown
1. Type "test@example.com" into the email field
   > Email field accepts input
```

**Version 3** (after discovering sub-steps were needed):

```markdown
1. Open the page
   - Click field 1
   - Click field 2
   > All fields are filled
```

Each change followed the same pattern:

1. Discuss the change
2. Update the spec
3. Update the parser/serializer
4. Update tests
5. Verify round-trip fidelity

The spec was never "done." It was always the current truth.

The PRD keeps the product consistent across time. But within a single session, consistency has another enemy: context noise.

## The Context Window Wall

Here's something nobody warns you about: Claude Opus with a million-token context window sounds infinite. It's not.

Around 50% context usage, the agent started getting noticeably worse. Not dramatically - it still compiled code and ran tests. But the judgment degraded. It would forget rules we'd established earlier in the session. It would re-introduce patterns we'd explicitly rejected. It would make the same fix three times in a row, each time slightly different, none of them working.

The ID editing bug was the clearest example. We'd already solved the identical problem for the title field - use one `BasicTextField` always, toggle `readOnly`, don't switch between composables. The agent knew this. It had written the solution. But when faced with the same problem for the ID field, it tried `remember`, then `rememberSaveable`, then `key()`, then timestamp guards, then `focusProperties` - everything except the approach that was already working twenty lines above.

A million tokens of context doesn't mean a million tokens of *useful* context. Past a certain point, the signal-to-noise ratio degrades. The agent has seen so many iterations, so many failed approaches, that it can't distinguish the final working solution from the twelve attempts that preceded it.

This is another reason the spec matters. When the agent's in-session memory gets noisy, the spec is the clean, authoritative source. "What does the ID field do? Read the spec." Not "scroll back through 400 messages of debugging."

## The Subagent Pattern

The spec helps when context is noisy. A better answer is to keep context clean in the first place - which is where subagents come in.

For larger changes, I used a pattern called Subagent-Driven Development. The controller (main agent) reads the plan, dispatches a fresh subagent per task, reviews the output, and moves to the next task.

The key insight: **subagents don't read the spec file.** The controller extracts exactly what they need and sends it as context. This means:

- Subagents get focused, relevant instructions
- No risk of a subagent misinterpreting spec sections meant for other tasks
- The controller maintains coherence across tasks

But it also means the controller must understand the spec deeply enough to brief each subagent correctly. Which brings us back to why the spec matters - it's the controller's source of truth.

## When Things Break

The most painful moment: I asked the agent to fix all the visual issues from a UX review, extract strings to a properties file, update the TestRunPanel design, and add Nice to Have polish. The agent dispatched subagents for all of it, made the changes, tests passed.

Then I ran the plugin and found that the title editing had a race condition. `onFocusChanged` fired before `onClick`, causing the edit mode to toggle on and off in the same frame. The fix took multiple attempts - delayed commits, timestamp guards, focus property hacks - before we found the real solution: use one `BasicTextField` always (with `readOnly` toggle) instead of switching between `Text` and `BasicTextField`.

The spec said "no layout shift." The implementation had layout shift. The spec was right; the code was wrong. Having the spec as the authority made it clear what "correct" meant, even when debugging Compose Desktop focus behavior at midnight.

## What I'd Do Differently

**1. Start with **`AGENTS.md`** on day zero.** Don't wait until you've already established bad patterns. The agent will follow whatever norms exist - or invent its own.

**2. Make the spec update the first step in every change, not the last.** "Spec first" sounds obvious, but when you're in the flow of "just fix this one thing," it's the easiest step to skip. Put it in `AGENTS.md` as a hard rule.

**3. Keep the spec focused.** Our spec grew to describe the current state, not the history of changes. No changelogs, no migration notes, no "we used to do X but now we do Y." Just what the product is right now.

**4. Use **`AGENTS.md`** for behavioral rules, not implementation details.** "Never hardcode strings" belongs in `AGENTS.md`. "The parser uses regex `^\d+\.\s*(.*)$`" does not - that's derivable from reading the code.

**5. Save feedback as memory.** When you correct the agent ("don't do X"), save it. The agent won't remember next session unless you tell it to.

## The Cursor That Wasn't There

Late in the session, I was clicking around the preview panel and something felt wrong. Not broken - just *off*. It took me a few seconds to notice: the mouse cursor wasn't changing to a pointer hand on interactive elements. Dropdowns, buttons, clickable text - all showing the default arrow.

This is the kind of thing a human developer notices immediately when using the product. And the kind of thing an AI agent will never catch on its own, because it doesn't hover over buttons.

The fix wasn't hard. But the *approach* matters.

My first instinct was to tell the agent "add hand cursor to all clickable elements." And the agent did exactly that - it replaced every `.clickable {}` with a version that included `pointerHoverIcon(Hand)`. Problem solved, right?

Wrong. Because `IconButton` (from Jewel) doesn't use our `.clickable` - it has its own click handler. Neither does `ListComboBox`. The agent had fixed the custom clickable areas but missed every framework component.

So we went systemic. One rule in `AGENTS.md`: "Never use bare `clickable` or `IconButton` in SpeQA UI code." Now every future agent session knows the pattern. Every new interactive element gets the cursor automatically - not because someone remembered, but because the only available tools do it by default.

This is the real lesson: **the best rules don't require enforcement. They're baked into the API.**

If you have a coding standard that depends on developers remembering to add something every time - it will fail. Wrap it in an abstraction. Make the right thing the default thing. Then put it in `AGENTS.md` so the agent uses the abstraction instead of the raw API.

The spec got updated too. A new bullet point under UX decisions: "All interactive elements must show a hand cursor on hover." Seven months from now, when someone (or some agent) adds a new button, the spec says it needs a pointer cursor, `AGENTS.md` says to use the specific method, and the wrapper adds it automatically.

Three layers of defense. Zero things to remember.

## The Hook That Saved the Spec

Remember that rule I mentioned at the top? "Update the spec first." The one in `AGENTS.md`. The one I saved to the agent's persistent memory. Twice.

It didn't work.

Not "didn't work sometimes." Didn't work *consistently*. I'd ask for a feature. Claude would start coding. I'd interrupt: "Did you update the spec?" And it would say some version of "Right, sorry" and go edit the spec retroactively. Then the next feature request, same thing. Code first, spec never.

I tried everything the platform gives you. `AGENTS.md` had the rule in bold. The agent's memory file had the rule saved with a "Why" and a "How to apply." I even saved a *second* memory entry that said, essentially, "No seriously, the first memory entry about this is important."

Still forgot.

Here's the thing about AI agents and rules: they're like New Year's resolutions. The agent genuinely intends to follow them. It reads them at the start of every session. It agrees they're important. And then the moment you say "add inline editing to the preview panel," all that good intention evaporates in the face of an interesting implementation problem.

So I stopped asking nicely. I asked the agent: how do we make sure you stop forgetting? It answered: *write a hook.* So I told it: *then write yourself one.*

It did. The setup is three files. The wiring goes in `.claude/settings.local.json`:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "bash .claude/hooks/post-spec-edit.sh"
        }]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "bash .claude/hooks/pre-kt-edit.sh",
          "statusMessage": "Checking spec-first workflow"
        }]
      }
    ]
  }
}
```

The `PostToolUse` script drops a marker file whenever the spec is touched:

```bash
# .claude/hooks/post-spec-edit.sh
FILE_PATH=$(cat /dev/stdin | jq -r '.tool_input.file_path // empty')
if echo "$FILE_PATH" | grep -q 'speqa-design'; then
  touch /tmp/.speqa-spec-edited
fi
```

The `PreToolUse` script blocks `.kt` edits unless the marker exists:

```bash
# .claude/hooks/pre-kt-edit.sh
FILE_PATH=$(cat /dev/stdin | jq -r '.tool_input.file_path // empty')
if echo "$FILE_PATH" | grep -q '\.kt$'; then
  if [ -f /tmp/.speqa-spec-edited ]; then
    exit 0
  else
    echo 'BLOCKED: Update the spec BEFORE editing .kt files.'
    exit 2
  fi
fi
```

Before every `Edit` or `Write` tool call, the pre-hook checks: are you touching a `.kt` file? If so, has the spec been touched in this session? No? **Blocked.** Exit code 2. The agent literally cannot write Kotlin until the spec is touched.

And then, in the very same turn it finished writing the hook, the agent tried to edit a `.kt` file - and got blocked by its own hook.

I wish I could say that moment was planned for dramatic effect. It wasn't. The agent had just created the enforcement mechanism, and in the very same turn, tried to violate the rule it was designed to enforce. The hook caught it. The agent backed up, edited the spec, then continued with the implementation.

That's the whole lesson right there.

`AGENTS.md` is a suggestion. Memory is a suggestion with persistence. But a `PreToolUse` hook is a *wall*. The agent can't charm its way past it, can't promise to "do it next time," can't get distracted by an interesting composable problem. The shell script doesn't care about context or intent. It checks a condition and blocks or allows.

This maps to something we already know from software engineering: **the best constraints are mechanical, not behavioral.** We don't ask developers to remember to run the linter - we put it in CI. We don't ask them to avoid committing secrets - we use pre-commit hooks. We don't ask the AI agent to update the spec - we make it physically impossible to skip.

But here's the twist: **the first hook wasn't strict enough.**

It checked whether the spec had been edited *at any point in the session*. Edit the spec once at the beginning, and every `.kt` file after that sailed through unchecked. The agent figured this out (not deliberately - it just happened). It would edit the spec for the first change, then make five more `.kt` changes without touching the spec again.

Here's the case that made that failure mode impossible to ignore.

### The bug fix that needed three spec edits

A pattern I hadn't expected: **bug fixes are more dangerous than features.**

With a new feature, the agent has to read the spec, understand the requirement, update the spec, then implement. The workflow is naturally sequential. But with a bug fix, the agent thinks it already knows what's wrong. The spec already describes the correct behavior. The code just doesn't match. So it skips the spec and dives straight into the code.

This is exactly when things spiral.

We had a chip input field - tags and environments. The spec said "clicking `+` opens a floating text field via `Popup` that overlays the content without pushing it down." The implementation rendered the input inline, pushing everything below it. Classic spec-code mismatch.

The agent fixed the wrong component first (`ChipField`), because it looked like the right one. But `ChipField` wasn't even used - the actual component was `TagCloud` in a completely different file. One wasted fix attempt.

Then it fixed `TagCloud` but the popup appeared to the right of the `+` button instead of below it. Another attempt needed.

The session-wide hook didn't help here - it had already been satisfied earlier in the day by a single spec edit. The agent fired off three rapid `.kt` attempts without once being forced back to the spec. Each of those "quick fixes" would have been caught if the hook had demanded a spec edit *per attempt* - not because the spec was wrong, but because the act of editing it would have forced the agent to articulate what it was about to try and why the last attempt didn't work. The difference between "let me try this" and "I understand that the previous approach failed because X, and the next approach addresses Y."

So I tightened the screw. The change was one line in `pre-kt-edit.sh` - the marker now gets consumed on each `.kt` pass instead of persisting:

```bash
# .claude/hooks/pre-kt-edit.sh
FILE_PATH=$(cat /dev/stdin | jq -r '.tool_input.file_path // empty')
if echo "$FILE_PATH" | grep -q '\.kt$'; then
  if [ -f /tmp/.speqa-spec-edited ]; then
    rm /tmp/.speqa-spec-edited   # one spec edit = one .kt edit
    exit 0
  else
    echo 'BLOCKED: Update the spec BEFORE editing .kt files.'
    exit 2
  fi
fi
```

After each spec edit, a marker file is created. Before each `.kt` edit, the marker is checked and **consumed**. One spec edit unlocks exactly one `.kt` edit. The next `.kt` edit needs another spec edit. No bulk passes.

And the `AGENTS.md` rule got upgraded too: "Even for bug fixes where the spec already describes correct behavior - document why the previous implementation attempt failed." Because if the spec is already right and the code is wrong, that's worth recording. What assumption was wrong? What edge case was missed? The spec becomes not just a description of the product, but a record of hard-won understanding.

The hierarchy looks like this:

1. **Documentation** (`AGENTS.md`) - the agent reads it, agrees, and sometimes follows it
2. **Memory** - the agent reads it, remembers it better, and sometimes follows it
3. **Hooks** - the agent has no choice
4. **Strict hooks** - one-shot tokens, not session-wide passes

You need all four. And you'll discover layer 4 the same way I did: by watching layer 3 fail in a way you didn't anticipate.

## When the PRD Becomes the Problem

There's a less convenient truth here too: if you push "spec first" hard enough, the spec can become its own kind of mess.

After a few hundred iterations, my PRD wasn't just the current product definition anymore. It was also carrying the residue of the process that produced it. Tiny clarifications. Micro-adjustments. Sentences the agent added because the hook forced it to touch the spec before touching a Kotlin file, even when the real work was clearly going to happen across five related files.

At some point, that stopped being discipline and turned into the agent editing the PRD just to earn the right to edit code.

I adjusted the workflow. Instead of updating the PRD before every single file edit, I switched to batch edits: touch the PRD first, then work through a related set of files while that context was still fresh. That was better. Much better, actually.

But it still didn't solve the underlying problem.

Over time the PRD grew to roughly 2,000 lines. It was still useful. It still contained real product knowledge. But it was also undeniably dirty. Not wrong. Just noisy. Too much of it existed because the workflow rewarded touching the document, not because the reader truly needed another sentence.

So I did something I feel conflicted about even now: I cleaned house.

I started keeping feature-specific PRDs for the work that was active right now. I simplified the commit history. I removed specs that were just sitting in the repository long after the implementation had stabilized. I tried to leave the repo with one clean description of the present instead of a museum of every intermediate thought.

That absolutely made the repository easier to work in.

It also absolutely destroyed some historically valuable context.

I know I threw away information that would have been useful later: why a dead-end was a dead-end, what tradeoffs we rejected, which weird edge case forced a design tweak three months ago. That's real knowledge. I don't think "just delete old specs" is the final answer.

It feels more like a temporary coping mechanism. A local optimization. Something that worked for me in this repo once the cost of carrying all that history inside the active PRD got too high.

The real solution probably needs to be more systemic than that. Maybe it's a product feature. Maybe it's a workflow pattern. Maybe it's a better separation between current truth and decision history so the working PRD stays sharp without erasing the path that got you there.

I don't have that answer yet.

I suspect that's a future post.

## The Bigger Picture

There's a pattern emerging in how we build software with AI agents. It's not "write a prompt, get code." It's more like:

1. **Spec** - What are we building? Write it down.
2. **Rules** - How do we build? Put it in `AGENTS.md`.
3. **Enforce** - Rules get ignored. Back them with hooks.
4. **Plan** - What are the tasks? Break it down.
5. **Execute** - Dispatch agents with precise context.
6. **Verify** - Does the code match the spec?
7. **Update** - Spec changed? Update it first. Always first.

The spec isn't documentation. It's the interface between you and the agent. It's how you maintain control when the agent is writing code faster than you can review it. It's how every new session with the agent - months later, after the product has changed - starts from the same shared understanding as the very first one.

The code is ephemeral. The spec is the source of truth.

And honestly? I don't think any of us has really figured out how to work with these agents yet.

Everything in this post - `AGENTS.md`, the memory entries, the hooks, the batch edits, the eventual cleanup - is a layer I added because the previous one wasn't enough on its own. Each piece does real work. None of them is the whole answer. What I've written down today as "the workflow" will almost certainly look naive a year from now.

That's the honest part. We're not at the end of figuring this out - we're at the very beginning. The patterns that will eventually feel obvious are being invented right now, in other people's repos, by other people hitting the same walls and writing different workarounds. This post is one data point, not a manual.

I'm sharing it because it worked for me, not because I think it's the only right way.
